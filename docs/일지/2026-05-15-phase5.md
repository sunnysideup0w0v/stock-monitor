### Phase 5.2 — 조건 스크리너 엔진

**`ScreenerCondition`** (Codable, Identifiable):

| 타입 | 설명 |
|------|------|
| `priceRange` | 현재가 범위 (원) |
| `volumeMin` | 최소 거래량 |
| `changeRateRange` | 등락률 % (시가 대비 종가) |
| `perRange` | PER 범위 |
| `pbrRange` | PBR 범위 |
| `marketCapRange` | 시가총액 범위 (억원, DB는 백만원 단위) |
| `sectorFilter` | 업종 (DB distinct 조회) |
| `marketFilter` | 시장 구분 (KOSPI / KOSDAQ) |

**`ScreenerEngine`** (`@unchecked Sendable`, 싱글턴):
- `run(conditions:limit:)` → GRDB `QueryInterfaceRequest`를 조건별로 동적 빌드
- `changeRateRange`는 SQL 식 `CAST(close - open AS REAL) * 100.0 / NULLIF(open, 0)` 사용
- PER/PBR 조건 시 `NULL` 행 자동 제외
- 결과: 시가총액 내림차순, 최대 300개

**`DatabaseManager` 추가 메서드**:
- `fetchStockUniverse(matching:)` — `QueryInterfaceRequest` 직접 실행
- `fetchDistinctValues(column:table:)` — Picker용 distinct 값 조회

### 버그 수정: KRX 업데이트 버튼 고착

**원인**: 버튼 액션이 `isFetching = true` 설정 후 `Task { await fetchAndStore() }`를 fire-and-forget으로 실행. `fetchAndStore()` 내부 guard가 조기 return하면 `.krxDataUpdated` 알림이 오지 않아 뷰의 `isFetching`이 `true`에 고착.

**수정**: Task 완료 후 `loadStats()`를 직접 호출해 `isFetching = KRXManager.shared.isFetching`으로 동기화. 성공/실패 상태 메시지(`statusMessage`) 추가.

```swift
// 수정 전
Button {
    isFetching = true
    Task { await KRXManager.shared.fetchAndStore() }  // awaited 아님
}

// 수정 후
private func fetch() {
    guard !isFetching else { return }
    isFetching = true
    Task {
        await KRXManager.shared.fetchAndStore()
        loadStats()           // Task 완료 후 직접 동기화
        if stockCount == 0 { statusMessage = "수신 실패 ..." }
        else { statusMessage = "✓ 업데이트 완료" }
    }
}
```

## 설계 결정

### KRX API 선택 이유
당초 `openapi.krx.co.kr`(회원가입 필요)을 검토했으나 `data.krx.co.kr` POST API가 API 키 없이 전종목 데이터를 제공한다는 점에서 선택. `Referer: https://data.krx.co.kr/` 헤더만 필요.

### ScreenerEngine을 actor가 아닌 @unchecked Sendable로 구현한 이유
`ScreenerEngine`은 mutable state가 없고 `DatabaseManager.fetchStockUniverse(matching:)`에만 의존하므로 actor 오버헤드가 불필요. 기존 `DatabaseManager`와 동일한 패턴(`@unchecked Sendable`)으로 일관성 유지.

### ClaudeAnalyzer를 actor로 구현한 이유
향후 요청 취소(`Task.cancel()`) 또는 in-flight 중복 방지를 위해 mutable state(`currentTask` 등)를 추가할 여지가 있기 때문. 현재는 stateless이지만 actor로 선언해 확장성 확보.

### AI 분석 모델을 claude-sonnet-4-5로 선택한 이유
사용자의 Anthropic API 요금 부담을 고려해 Opus 대신 Sonnet 선택. 주식 스크리닝 분석 정도의 태스크는 Sonnet으로 충분.
